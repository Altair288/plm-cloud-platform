PLM 系统租户化元数据管理设计文档
1. 业务背景与设计理念
本系统采用 “元数据驱动业务（Metadata-Driven）” 的设计模式。为了兼顾行业标准化与企业个性化，系统在物理层面将“标准元数据”与“企业业务数据”进行彻底隔离。

plm_meta (元数据库)：存放行业标准、无限层级分类模板、通用属性池等公共资源。

plm_runtime (运行数据库)：存放各租户（企业）抽取后的实例化数据、企业私有规则及特有物料数据。

2. 原始需求定义 (Original Requirements)
2.1 租户化组织架构
组/租户模式：用户注册后自动创建一个“租户（Tenant）”。租户是数据隔离的最小单位。

多视角切换：

平台管理员：负责维护 plm_meta，审核租户的回写申请。

租户管理员：负责本租户内的成员管理、元数据抽取、编码规则自定义。

租户成员：负责具体的业务执行（如导入导出、创建物料）。

2.2 元数据生命周期：抽取、扩展与回写
抽取 (Adopt)：租户从 plm_meta 中勾选所需的分类和属性，实例化到 plm_runtime。

自定义扩展：租户在 plm_runtime 中基于自身业务修改编码规则、新增企业私有属性。

审批回写 (Push-back)：租户特有数据若具备通用性，可发起回写申请。通过平台管理员审批后，同步至 plm_meta。

3. 权限管控细节 (基于 Sa-Token 实现)
系统采用 Multi-Tenant RBAC (多租户角色访问控制) 模型，结合 Sa-Token 的多账号体系实现物理与逻辑的双重隔离。

3.1 账号体系设计
通过 Sa-Token 的 StpUtil 区分两套认证域：

TYPE_USER：针对租户端用户，Token 中强制绑定 tenant_id。

TYPE_ADMIN：针对平台端管理员，拥有跨租户审计权限。

3.2 核心权限矩阵
权限标识 (Permission)	描述	归属视角	作用域 (Schema)
meta:read	查看元数据模板	全体	plm_meta
meta:manage	修改/发布元数据	平台管理员	plm_meta
runtime:config	配置租户编码规则	租户管理员	plm_runtime
runtime:operate	物料创建/导入/导出	租户成员	plm_runtime
workflow:apply	发起回写审批	租户管理员	跨库操作
workflow:approve	审核回写申请	平台管理员	跨库操作
3.3 数据隔离实现逻辑
Token 载荷：Sa-Token 在登录成功后，将 tenant_id 存入 Session。

自动注入：通过 Sa-Token 的注解或拦截器，在所有 plm_runtime 的 SQL 请求中强制追加 AND tenant_id = #{current_tenant_id}。

多账号鉴权示例：

租户端登录：StpUser.login(id)

平台端登录：StpAdmin.login(id)

接口拦截：使用 @SaCheckRole("tenant_admin") 确保租户管理员才能配置私有规则。

4. 关键业务流程设计
4.1 租户自定义规则流程
租户管理员进入“编码规则设计”页面。

后端根据当前 Token 中的 tenant_id 从 plm_runtime.tenant_config 中检索规则。

若该租户未配置，则 Fallback（回退）至 plm_meta 的默认示例规则。

用户修改并保存，数据写入 plm_runtime，该租户后续的 derive 接口将优先采用此私有规则。

4.2 审批回写逻辑流程
数据标记：租户在 plm_runtime 中新增数据，标记为 local 状态。

发起申请：租户管理员调用申请接口，Sa-Token 校验 workflow:apply 权限。

锁定状态：数据进入 pending_approval 状态，不可再次修改。

平台审核：平台管理员在后台查看到申请，Sa-Token 校验 workflow:approve 权限。

数据迁移：审批通过，后端开启跨库事务，将数据同步至 plm_meta，并通知所有租户有新版本更新。

5. 扩展性考虑
分库支持：由于物理分库，Sa-Token 的持久化存储（如 Redis）需支持按租户隔离，防止 Token 混淆。

PLM 租户化用户管理与元数据协同需求规范书
1. 组织与账号模型需求 (Organization & Account)
1.1 租户与组空间
租户实体化：系统必须支持“租户（Tenant）”概念，注册即创建独立业务空间。

租户自管理：租户管理员可自主邀请成员、禁用账号，且操作仅在本租户内生效。

多身份支持：支持同一用户（Email/手机号）在不同租户中拥有不同身份（如在 A 租户是管理员，在 B 租户是普通成员）。

1.2 视角隔离
平台管理视角：具备全局视角，负责 Meta 库维护、租户入驻审核、全局资源监控。

租户业务视角：仅可见本租户数据及 Meta 库下发的标准数据，无法感知其他租户的存在。

2. 权限管控需求 (Access Control Details)
2.1 基于 Sa-Token 的鉴权范围
登录认证 (Authentication)：

支持不同登录终端的 Token 隔离（如：平台管理后台与租户操作端使用不同的 Cookie 名称或 Header）。

支持 Session 强踢与多地登录限制。

功能鉴权 (Authorization)：

角色模板化：平台预设角色（超级管理员、租户管理员、研发、采购），租户可基于模板微调权限。

按钮级控制：导出、回写申请、规则修改等核心操作必须绑定特定的 Permission Key。

2.2 数据安全隔离范围
物理/逻辑双隔离：所有 API 请求必须经过租户过滤器，确保 SQL 自动带入 tenant_id。

跨库访问受控：从 runtime 库向 meta 库的数据回写必须经过提权认证，禁止租户端直接执行 Meta 库的 Update/Insert。

3. 元数据协同与回写需求 (Metadata Synergy)
3.1 抽取与实例化需求
按需认领：租户可从 Meta 库中选择特定的无限层级分类树分支进行同步。

版本追溯：当 Meta 库更新时（如行业标准变更），需在租户端提供“版本差异对比”及“一键升级/忽略”功能。

3.2 自定义规则需求
规则覆盖逻辑：租户可针对已抽取的分类，自定义编码生成规则。该规则存储在 Runtime 库，不回写 Meta。

私有属性扩展：支持租户在标准分类下新增仅本企业可见的属性字段。

3.3 审批回写需求 (Push-back Workflow)
回写申请触发：租户管理员可将标记为“通用”的私有数据发起入库申请。

预校验逻辑：发起回写前，系统必须自动执行：唯一性校验（编码是否已被 Meta 占用）、格式合规校验。

审批生命周期：包含 待提交 -> 平台审核中 -> 退回修改 -> 审核通过/同步完成。

4. 设计边界定义 (Design Scope)
4.1 在设计范围内的 (In-Scope)
租户生命周期管理：创建、冻结、数据初始化。

RBAC 权限模型：用户-角色-权限三要素管理。

多视角路由切换：根据登录态自动分发 UI 界面。

跨库同步逻辑：Meta 与 Runtime 之间的数据传输机制。

审批流状态机：处理回写申请的流程节点流转。

4.2 不在设计范围内的 (Out-of-Scope)
租户间的直接协同：不涉及租户 A 与租户 B 之间的数据共享。

复杂的外部 IDP 集成：暂不考虑对接 LDAP 或 OAuth2（以 Sa-Token 自有认证为主）。

业务数据（物料）的审批：本次仅涉及元数据（分类、属性、规则）的回写审批。
